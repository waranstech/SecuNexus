API Security Testing — Beyond OWASP Top 10

SecuNexa Team | Meteonic Innovation Pvt Ltd
Share:

APIs have become the primary attack surface for modern applications. As organizations shift from monolithic architectures to microservices, and as mobile applications and third-party integrations proliferate, the number of API endpoints exposed by a typical enterprise has grown exponentially. Yet API security testing remains significantly less mature than traditional web application security testing. Most organizations either skip API-specific testing entirely or limit their efforts to checking the OWASP API Security Top 10 without going deeper into the nuanced vulnerabilities that APIs introduce.

The OWASP API Security Top 10 Explained

The OWASP API Security Top 10 provides a foundational framework for understanding the most common and critical API vulnerabilities. Every API security testing program should begin here, but should not stop here.

API1: Broken Object Level Authorization (BOLA)

BOLA occurs when an API endpoint accepts an object identifier from the client and does not verify that the authenticated user has permission to access the referenced object. An attacker can enumerate object IDs to access other users' data. This is the most prevalent API vulnerability because developers often assume that if a user is authenticated, they are authorized to access any object. Testing for BOLA requires sending requests with object IDs belonging to other users and verifying that access is denied.

API2: Broken Authentication

API authentication vulnerabilities include weak token generation, missing token expiration, credential stuffing susceptibility, and improper token validation. Unlike web applications where session management is handled by the framework, APIs often implement custom authentication schemes that are prone to implementation errors. Testing should verify token entropy, expiration enforcement, revocation mechanisms, and resistance to brute-force attacks.

API3: Broken Object Property Level Authorization

This vulnerability occurs when an API exposes object properties that the user should not be able to read or modify. Mass assignment attacks exploit endpoints that accept JSON objects and blindly map properties to internal data models, allowing attackers to set fields like isAdmin or accountBalance. Testing requires sending requests with additional properties beyond what the UI normally sends and checking whether they are accepted.

API4: Unrestricted Resource Consumption

APIs that do not limit resource consumption are vulnerable to denial-of-service attacks. This includes missing rate limiting, no pagination limits on list endpoints, allowing oversized request bodies, and enabling expensive queries. A single crafted request to an unprotected endpoint can consume enough server resources to degrade service for all users.

API5: Broken Function Level Authorization

This vulnerability occurs when administrative functions are accessible to regular users. APIs often implement role-based access at the UI level but fail to enforce it at the API layer. An attacker who discovers the administrative API endpoint (through documentation, JavaScript analysis, or endpoint enumeration) can invoke privileged functions. Testing requires attempting to call administrative endpoints with non-administrative credentials.

API6-10: Additional Critical Risks

The remaining categories cover unrestricted access to sensitive business flows, server-side request forgery (SSRF), security misconfiguration, improper inventory management, and unsafe consumption of APIs. Each requires specific testing techniques that go beyond simple vulnerability scanning.

Beyond OWASP: Advanced API Security Testing

Business Logic Testing

Business logic vulnerabilities in APIs are among the most damaging and hardest to detect because they involve legitimate API operations used in unintended sequences or combinations. Examples include applying discount codes multiple times, transferring negative amounts to reverse fund flows, skipping required workflow steps by calling endpoints out of sequence, and exploiting race conditions in concurrent requests.

Automated scanners cannot detect business logic flaws because they require understanding of the application's intended behavior. Effective testing combines manual analysis of API documentation and source code with targeted automated tests that verify business rule enforcement. Test cases should cover multi-step workflows, boundary conditions on business-critical values, and concurrent request scenarios.

Authentication Deep Dive

Beyond the basic OWASP authentication checks, comprehensive API authentication testing should cover JWT implementation weaknesses (algorithm confusion attacks, missing signature verification, key leakage), OAuth flow vulnerabilities (open redirects, state parameter bypasses, scope escalation), API key management issues (keys in URLs, missing rotation, excessive permissions), and multi-factor authentication bypass through API endpoints that do not enforce MFA even when the web UI does.

Rate Limiting and Abuse Prevention

Effective rate limiting testing goes beyond confirming that a rate limit exists. Test whether rate limits can be bypassed by changing IP addresses (proxy rotation), modifying request headers (X-Forwarded-For manipulation), using different API keys, or making slight variations to the request. Verify that rate limits are applied per-user rather than globally, and that they cover all relevant dimensions including login attempts, password reset requests, resource creation, and data export endpoints.

API Discovery and Shadow APIs

Shadow APIs are endpoints that exist in the production environment but are not documented or managed by the security team. They arise from incomplete decommissioning of old API versions, development and debugging endpoints left active, automatically generated API endpoints from frameworks, and third-party integrations that create undocumented callback endpoints.

API discovery testing involves crawling the application's JavaScript bundles for hardcoded API URLs, analyzing network traffic during normal application usage, brute-forcing common API path patterns, examining API documentation (Swagger/OpenAPI files) for hidden or deprecated endpoints, and checking for version prefixes that indicate older API versions still active in production.

GraphQL Security

GraphQL introduces unique security challenges that REST-focused testing tools often miss entirely. Introspection queries can reveal the entire API schema to attackers if not disabled in production. Deeply nested queries can cause denial of service by creating exponential resolution complexity. Batch queries can be used to bypass rate limits by combining thousands of operations in a single request. Field-level authorization is difficult to implement correctly because GraphQL allows clients to request arbitrary field combinations.

Testing GraphQL APIs requires schema analysis to identify sensitive fields and operations, query complexity analysis to find denial-of-service vectors, authorization testing on individual fields and type resolvers, and verification that introspection is disabled or restricted in production environments.

Data Exposure and Serialization

APIs frequently expose more data than necessary in responses. Verbose error messages may reveal stack traces, database schema details, or internal service topology. List endpoints may return full objects when summaries would suffice. Debug headers or metadata may be present in production responses. Testing should compare API responses against the minimum data required by the consuming client and flag any excessive exposure.

Serialization vulnerabilities occur when APIs accept serialized objects (JSON, XML, Protocol Buffers) and deserialize them without proper validation. Insecure deserialization can lead to remote code execution, particularly in Java and .NET environments where type information is included in the serialized data.

Building an API Security Testing Program

An effective API security testing program integrates multiple approaches across the development lifecycle:

  1. Design review: Evaluate API designs for security before implementation begins. Check authentication models, authorization granularity, data exposure, and rate limiting strategy.
  2. Static analysis: Use SAST tools to analyze API controller code for injection vulnerabilities, missing authorization checks, and insecure data handling.
  3. Automated scanning: Run DAST tools configured for API testing against staging environments. Provide OpenAPI/Swagger specifications to ensure complete endpoint coverage.
  4. Manual testing: Conduct manual penetration testing focused on business logic, authentication flows, and authorization enforcement that automated tools cannot assess.
  5. Runtime protection: Deploy RASP or API gateway security to detect and block attacks in production, providing a safety net for vulnerabilities missed during testing.

How SecuShield Protects APIs

SecuShield provides runtime API protection through deep integration with the application runtime.
