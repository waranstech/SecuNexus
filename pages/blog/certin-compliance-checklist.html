CERT-In Compliance Checklist for Application Security Teams

SecuNexa Team | Meteonic Innovation Pvt Ltd
Share:

The Indian Computer Emergency Response Team (CERT-In), operating under the Ministry of Electronics and Information Technology, has issued directives that create binding obligations for organizations operating in India. The April 2022 directive and subsequent clarifications established some of the most stringent incident reporting and cybersecurity requirements globally. For application security teams, these directives translate into specific technical and procedural requirements that must be implemented, documented, and regularly demonstrated.

This article provides a practical 21-point checklist that application security teams can use to assess and improve their CERT-In compliance posture.

CERT-In Directive Overview

The CERT-In directive of April 2022 applies to all service providers, intermediaries, data centers, body corporates, and government organizations operating in India. Key provisions relevant to application security include mandatory reporting of cyber incidents within six hours of detection, maintenance of ICT system logs for 180 days within Indian jurisdiction, synchronization of ICT system clocks with NTP servers, designation of a point of contact for CERT-In communication, and maintenance of comprehensive records of ICT infrastructure and assets.

While the directive covers broad cybersecurity operations, application security teams play a critical role in several compliance areas. The six-hour reporting requirement means that organizations must be able to rapidly determine the impact of application-level vulnerabilities. The logging requirements apply to application logs. And the asset inventory requirements extend to software components and dependencies.

The 21-Point Application Security Checklist

Incident Preparedness (Points 1-5)

1. Six-Hour Incident Detection Capability
Verify that your application monitoring can detect security incidents (data breaches, unauthorized access, application compromise) and generate alerts within a timeframe that allows six-hour reporting to CERT-In. This requires real-time application security monitoring, not just periodic scanning.

2. Application-Level Incident Response Playbook
Maintain documented playbooks for common application security incidents: SQL injection exploitation, authentication bypass, data exfiltration through API abuse, and supply chain compromise through vulnerable dependencies. Each playbook should include detection criteria, containment steps, evidence preservation procedures, and CERT-In reporting templates.

3. Vulnerability Impact Assessment Process
Establish a process to rapidly assess whether a newly disclosed vulnerability (such as a critical CVE in an open-source component) affects your applications. This requires maintaining current SBOMs and having the ability to query them programmatically. The faster you can determine impact, the faster you can make informed reporting decisions.

4. Point of Contact Designation
Designate and register a point of contact with CERT-In who has authority to report incidents and access to application security information. Ensure backup contacts are designated for coverage during leave or unavailability. The point of contact should have direct communication channels with the application security team.

5. Communication Templates
Prepare pre-formatted incident report templates that comply with CERT-In reporting requirements. Include fields for incident type, affected systems, timeline, impact assessment, and containment measures. Having templates ready eliminates formatting delays during the critical six-hour reporting window.

Application Logging and Monitoring (Points 6-10)

6. Application Log Retention (180 Days)
Configure all application logging to retain records for a minimum of 180 rolling days within Indian jurisdiction. This includes authentication logs, authorization decisions, API access logs, error logs, and security event logs. Verify that log storage is adequate for 180 days of production volume.

7. Clock Synchronization
Ensure all application servers synchronize their clocks with NTP servers designated by CERT-In or the National Informatics Centre (NIC). Accurate timestamps are essential for incident investigation and correlation. Configure NTP synchronization monitoring to detect clock drift.

8. Security Event Logging
Implement comprehensive security event logging in all applications. Log failed authentication attempts, privilege escalation events, access to sensitive data, API rate limit violations, input validation failures, and session management events. Each log entry must include timestamp, source IP, user identity (if available), and event description.

9. Log Integrity Protection
Implement controls to prevent log tampering. This may include write-once log storage, log forwarding to a centralized SIEM, cryptographic log chaining, or dedicated log management systems with access controls that prevent application administrators from modifying logs.

10. Monitoring and Alerting
Deploy real-time monitoring and alerting for application security events. Configure alerts for patterns that may indicate active exploitation: unusual authentication failure rates, unexpected API usage patterns, data access anomalies, and application error rate spikes. Alert thresholds should be tuned to minimize false positives while ensuring that actual incidents trigger notification within the timeframe needed for six-hour reporting.

Vulnerability Management (Points 11-15)

11. Continuous SAST Integration
Integrate static application security testing into the CI/CD pipeline so that every code change is analyzed for security vulnerabilities before deployment. Maintain quality gates that prevent high-severity vulnerabilities from reaching production. Document SAST tool configuration, rule sets, and quality gate thresholds.

12. Regular DAST Scanning
Conduct dynamic application security testing against all internet-facing applications on a scheduled basis (monthly at minimum for critical applications). DAST results should be tracked in the vulnerability management system with remediation SLAs. Maintain evidence of regular scanning cadence.

13. Software Composition Analysis
Run SCA scans on all applications to identify known vulnerabilities in open-source and third-party components. Maintain a current vulnerability database and scan continuously for newly disclosed CVEs affecting deployed applications. This capability is essential for rapid impact assessment when CERT-In issues advisories about specific vulnerabilities.

14. Remediation SLA Enforcement
Define and enforce remediation SLAs based on vulnerability severity: Critical (7 days), High (30 days), Medium (90 days), Low (180 days). Track actual remediation times against SLAs and escalate overdue items. Maintain historical data showing compliance with SLA targets.

15. Vulnerability Reporting to CERT-In
Establish a process for reporting application security vulnerabilities to CERT-In when required. This includes vulnerabilities that may affect other organizations (such as vulnerabilities in commonly used software that your organization develops or maintains). Maintain awareness of CERT-In advisory categories that trigger mandatory reporting.

Software Supply Chain (Points 16-19)

16. Software Bill of Materials (SBOM)
Generate and maintain SBOMs for all critical applications. SBOMs should be in standardized formats (CycloneDX or SPDX) and include all direct and transitive dependencies with version information. Update SBOMs with every build and maintain historical versions.

17. Dependency Update Discipline
Establish a regular cadence for reviewing and updating application dependencies. Track the age and vulnerability status of all dependencies. Prioritize updates for components with known vulnerabilities but also maintain awareness of components that are end-of-life or unmaintained.

18. Third-Party Risk Assessment
Conduct risk assessments for all significant third-party software components and services used in critical applications. Assess the vendor's security practices, update frequency, vulnerability response history, and license terms. Document the risk assessment and review it annually or when significant changes occur.

19. Secure Software Procurement
Implement security requirements in software procurement processes. Require vendors to provide SBOMs, vulnerability disclosure policies, and security testing evidence. For custom-developed software from vendors, require adherence to secure development lifecycle practices and evidence of security testing.

Governance and Documentation (Points 20-21)

20. Application Security Policy
Maintain a documented application security policy that covers secure development practices, security testing requirements, vulnerability management procedures, third-party component management, and incident response. Review and update the policy annually. Ensure all development teams are trained on the policy.

21. Compliance Evidence Repository
Maintain a centralized repository of all compliance evidence: scan reports, remediation records, policy documents, training records, incident response records, and audit findings. This repository should support rapid retrieval during CERT-In inquiries or regulatory audits. Implement retention policies that maintain evidence for the required period.
