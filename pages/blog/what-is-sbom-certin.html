What is SBOM and Why Does CERT-In Require It?

SecuNexa Security Research Team
Share:

Every modern software application is built on a foundation of third-party and open-source components. A typical enterprise application may contain hundreds or even thousands of open-source libraries, each with their own dependencies, vulnerabilities, and licensing obligations. A Software Bill of Materials (SBOM) is the definitive inventory of every component in your software, providing the transparency necessary for effective vulnerability management, license compliance, and regulatory adherence.

In the Indian context, CERT-In (Indian Computer Emergency Response Team) has increasingly emphasized the importance of software inventory management as part of its cybersecurity directives. Understanding what SBOM is, why it matters, and how to generate accurate SBOMs is now essential for every Indian organization that develops or deploys software.

Understanding Software Bill of Materials

An SBOM is a structured, machine-readable document that lists every component present in a piece of software. Think of it as a detailed ingredient label for software. Just as food packaging lists every ingredient, preservative, and additive, an SBOM lists every library, framework, module, and component that makes up an application.

A comprehensive SBOM includes several categories of information for each component. The component identity includes the name, version, supplier or author, and a unique identifier such as a Package URL (purl). Relationship data describes how components relate to each other, distinguishing between direct dependencies that your code explicitly imports and transitive dependencies pulled in by those direct dependencies. Licensing information documents the license under which each component is distributed, which determines what obligations your organization has when using that component. Vulnerability status maps each component to known security vulnerabilities (CVEs) in vulnerability databases.

The depth and accuracy of an SBOM depends entirely on the tools and methods used to generate it. A shallow SBOM generated from package manager lock files captures only declared dependencies. A deep SBOM generated through comprehensive SCA analysis also identifies vendored code (source files copied directly into the project), modified forks of open-source libraries, embedded components that bypass package managers, and transitive dependencies across multiple ecosystem boundaries.

Why CERT-In Requires SBOM

CERT-In's cybersecurity directives have evolved to address the growing threat landscape associated with software supply chain attacks. The rationale for SBOM requirements stems from several critical security concerns.

First, vulnerability response speed depends on knowing what you have. When a critical vulnerability like Log4Shell (CVE-2021-44228) is disclosed, organizations need to immediately determine whether they are affected. Without an SBOM, this determination requires manual auditing of every application, a process that can take days or weeks. With accurate SBOMs, the same determination takes minutes: search the SBOM inventory for the affected component and version, and you immediately know which applications need remediation.

Second, supply chain attacks are escalating in frequency and sophistication. Attackers are increasingly targeting open-source package repositories, compromising popular libraries to inject malicious code that propagates to downstream consumers. An SBOM provides the visibility needed to detect when a compromised component enters your software supply chain, enabling rapid response before the malicious code reaches production.

Third, CERT-In's six-hour incident reporting requirement makes software inventory visibility operationally critical. When a cybersecurity incident occurs, organizations must report to CERT-In within six hours, including details about affected systems and software. Without pre-existing SBOMs, gathering this information within the reporting window is practically impossible.

Fourth, CERT-In aligns with global trends toward mandatory SBOM requirements. The United States Executive Order 14028 on Improving the Nation's Cybersecurity established SBOM requirements for software sold to the federal government. The European Union's Cyber Resilience Act includes SBOM mandates for products sold in the EU market. India's regulatory direction follows this global trajectory, positioning SBOM as a fundamental cybersecurity hygiene practice.

SBOM Formats: CycloneDX vs SPDX

Two primary formats dominate the SBOM landscape, each with distinct characteristics and strengths.

CycloneDX is a lightweight SBOM format developed by OWASP (Open Worldwide Application Security Project). It was designed specifically for security use cases, making it particularly well-suited for vulnerability management workflows. CycloneDX supports JSON and XML encoding, includes native support for vulnerability references (VEX - Vulnerability Exploitability eXchange), and provides extensions for service definitions and hardware components. Its security-first design makes it the preferred format for application security teams. CycloneDX version 1.5 and later includes comprehensive support for dependency graphs, formulation data (how software was built), and licensing information.

SPDX (Software Package Data Exchange) is an ISO/IEC 5962:2021 international standard maintained by the Linux Foundation. SPDX has a broader scope than CycloneDX, covering detailed license information, file-level analysis, and inter-document relationships. It supports multiple serialization formats including JSON, XML, RDF, YAML, and tag-value. SPDX's strength lies in license compliance analysis, making it the preferred format for legal and procurement teams. The standardized format provides a common language for communicating software composition across organizational boundaries.

For Indian enterprises, the recommendation is to support both formats. Use CycloneDX for security-focused workflows such as vulnerability tracking and incident response. Use SPDX for license compliance analysis and legal documentation. Most modern SCA tools can generate both formats from the same analysis data.

How to Generate Accurate SBOMs

Generating an accurate SBOM requires more than running a single tool against your package manager lock file. A reliable SBOM generation process involves several stages and considerations.

Start with package manager analysis. Every application should have its declared dependencies analyzed through the appropriate package manager. For Java applications, analyze pom.xml or build.gradle files and resolve the complete dependency tree. For Node.js, analyze package-lock.json or yarn.lock. For Python, analyze requirements.txt, Pipfile.lock, or pyproject.toml. For .NET, analyze packages.config or project files with PackageReference entries. This step captures declared dependencies but misses vendored and embedded components.

Add binary analysis for compiled artifacts. Applications deployed as compiled binaries, container images, or packaged archives need binary-level analysis to identify components that are not visible in source-level package manager files. This is particularly important for C/C++ applications where dependency management is less standardized and components are frequently statically linked.

Implement source-level forensic analysis for the most comprehensive results. Forensic analysis examines the actual source code to identify components that were copied, vendored, or embedded without using a package manager. This catches code snippets from Stack Overflow, copied utility libraries, and modified forks of open-source projects that would otherwise be invisible in the SBOM.

Automate SBOM generation in your CI/CD pipeline. SBOMs should be generated automatically with every build, not as a periodic manual exercise. Integrate your SCA tool into the build pipeline so that an up-to-date SBOM is produced alongside every release artifact. Store SBOMs in a central repository alongside the software artifacts they describe, with version tracking that correlates SBOM versions to software versions.

Validate and enrich SBOM data. Raw SBOM output from automated tools should be validated for completeness and enriched with additional context. Verify that all components have valid package URLs or identifiers. Confirm that license information is accurate (this is where forensic license detection becomes valuable). Map components to known vulnerabilities and update the vulnerability status regularly. Add organizational metadata such as the application name, version, deployment environment, and data classification.

SBOM Management Best Practices

Generating SBOMs is only the first step. Effective SBOM management requires ongoing processes and infrastructure. Maintain a central SBOM repository where all application SBOMs are stored, versioned, and searchable. This repository should support querying across all SBOMs to answer questions like "which applications use this component?" when a new vulnerability is disclosed.

Implement continuous vulnerability monitoring against your SBOM inventory. As new CVEs are published, automatically cross-reference them against your SBOM repository and generate alerts for affected applications. This transforms SBOM from a static document into a dynamic security intelligence tool.

Establish retention policies that maintain historical SBOMs for at least the support lifetime of each software version. If a vulnerability is discovered in a component, you need the ability to check historical SBOMs to determine whether currently deployed or recently decommissioned software versions are affected.
