How to Build a DevSecOps Pipeline in Air-Gapped Environments

SecuNexa Team | Meteonic Innovation Pvt Ltd
Share:

Air-gapped environments present unique challenges for DevSecOps adoption. Organizations in defense, government, critical infrastructure, and regulated financial services operate networks that are physically isolated from the internet. This isolation is a security requirement, not a limitation to be circumvented. Yet modern DevSecOps practices assume constant internet connectivity for vulnerability database updates, package downloads, tool licensing, and cloud-based scanning services. Building an effective DevSecOps pipeline in an air-gapped environment requires rethinking every component of the toolchain.

The Air-Gapped Challenge

The fundamental challenge in air-gapped DevSecOps is that security tools depend on frequently updated external data sources. SAST tools need updated vulnerability patterns and rules. SCA tools require current vulnerability databases mapping CVEs to component versions. DAST tools depend on attack payload libraries and vulnerability signatures. Without internet access, these data sources become stale, reducing the effectiveness of security testing over time.

Beyond vulnerability data, the CI/CD pipeline itself faces challenges. Package managers cannot download dependencies from public repositories. Container registries cannot pull base images. Build tools cannot fetch plugins. License servers cannot validate tool activations. Every component that assumes network connectivity must be replaced with an offline alternative or adapted for air-gapped operation.

Setting Up the CI/CD Foundation

Internal Package Repositories

The first infrastructure component is an internal package repository that mirrors the public repositories your development teams depend on. Tools like Artifactory, Nexus Repository, or Verdaccio (for npm) can serve as internal proxies. In an air-gapped environment, these repositories are populated through a controlled import process rather than live mirroring.

Establish a transfer station, a system that sits at the boundary between the connected and air-gapped networks. This station downloads packages on the connected side, subjects them to security scanning, and transfers approved packages to the internal repository through a controlled media transfer process (typically using write-once media like DVDs or hardware-encrypted USB devices with chain-of-custody tracking).

Internal Container Registry

Container-based CI/CD requires an internal container registry (Harbor, Docker Registry, or GitLab Container Registry running on-premises). Base images must be imported through the same controlled transfer process as packages. Establish a golden image program where approved base images are maintained, scanned, and versioned internally. Development teams build FROM these internal images rather than pulling from Docker Hub.

Source Control and CI Server

Self-hosted Git servers (GitLab, Gitea, or Bitbucket Server) and CI servers (Jenkins, GitLab CI, or TeamCity) must run entirely within the air-gapped network. All plugins, extensions, and integrations must be pre-installed during initial deployment and updated through the controlled transfer process. Configure the CI server to reject any build step that attempts external network access as a safety measure.

Integrating SAST in Air-Gapped Pipelines

SAST tools in air-gapped environments must operate without cloud connectivity for analysis, rule updates, or license validation. This eliminates most SaaS-based SAST solutions and many commercial tools that require periodic license server check-ins.

Select a SAST tool that supports fully offline operation with perpetual or air-gapped licensing. The tool's rule database and vulnerability patterns should be distributable as update packages that can be imported through the controlled transfer process. Configure the SAST scanner as a CI pipeline stage that runs on every commit or merge request.

Pipeline integration should follow this pattern:

  1. Developer pushes code to the internal Git repository
  2. CI server triggers the build pipeline
  3. SAST scanner analyzes the codebase using locally installed rules and patterns
  4. Results are compared against the project's quality gate thresholds
  5. Build passes or fails based on vulnerability severity and count
  6. Results are stored in the internal vulnerability management dashboard

Integrating SCA Without Internet

SCA in air-gapped environments faces the most significant data freshness challenge because vulnerability databases are updated daily with newly disclosed CVEs. An SCA tool that cannot update its database becomes progressively less effective over time.

Offline Vulnerability Database Management

The SCA tool must support offline database updates through importable data packages. Establish a schedule for vulnerability database transfers, balancing security requirements (more frequent updates catch new vulnerabilities sooner) against operational overhead (each transfer requires the controlled media process). Weekly updates represent a reasonable baseline for most organizations; critical infrastructure may warrant daily transfers.

The database update process should include:

  • Download the vulnerability database update on the connected network
  • Verify the integrity and authenticity of the update package (cryptographic signatures)
  • Transfer the package through the approved media transfer process
  • Import the update into the air-gapped SCA tool
  • Verify successful import by checking the database version and entry count
  • Re-scan all monitored projects against the updated database to identify newly disclosed vulnerabilities in existing deployments

SBOM Generation and Management

SBOM generation should be an automated part of the build pipeline. Every build should produce an SBOM in CycloneDX or SPDX format alongside the build artifacts. These SBOMs serve as the definitive record of what components are deployed and enable rapid impact assessment when new vulnerabilities are disclosed. Store SBOMs in a centralized repository with version history to track how the application's dependency profile changes over time.

Integrating DAST in Isolated Networks

DAST integration requires deploying the scanner within the air-gapped network with access to the application's staging or test environment. The scanner must operate without cloud-based analysis services and maintain its attack payload library locally.

Configure the DAST scanner to run against the deployed application after build and deployment stages complete. Provide authentication credentials for the scanner to test authenticated endpoints. For API testing, import OpenAPI/Swagger specifications to ensure complete endpoint coverage. Schedule full DAST scans for off-peak hours to avoid impacting test environment performance, while running targeted scans (focusing on modified endpoints) as part of the regular CI pipeline.

Vulnerability Management Without Cloud Services

A centralized vulnerability management platform running within the air-gapped network aggregates findings from SAST, SCA, and DAST tools. This platform provides the unified dashboard, policy engine, and remediation tracking that cloud-based alternatives offer, but operating entirely on-premises.

Key capabilities for the vulnerability management platform include:

  • Finding correlation: Deduplication of vulnerabilities reported by multiple tools
  • Risk prioritization: Severity scoring that considers exploitability, data sensitivity, and deployment context
  • Remediation tracking: Assignment, SLA monitoring, and verification workflows
  • Compliance reporting: Pre-built reports for SEBI CSCRF, RBI CSF, CERT-In, and other frameworks
  • Audit trail: Complete history of scans, findings, and remediation actions for regulatory audits

Update and Maintenance Workflows

Maintaining a DevSecOps pipeline in an air-gapped environment requires disciplined update workflows. Each component of the toolchain, from the CI server to the security scanners to the vulnerability databases, needs periodic updates. Establish a regular cadence for each component:

  • Vulnerability databases: Weekly (or more frequently for critical sectors)
  • Security tool versions: Quarterly, aligned with vendor release cycles
  • CI/CD platform and plugins: Semi-annually, with emergency updates for critical patches
  • Base container images: Monthly, with immediate updates for critical OS patches
  • Package repository mirrors: As needed based on development requirements

Each update follows the same controlled transfer process: download on the connected network, verify integrity, scan for tampering, transfer through approved media, import and validate on the air-gapped network. Automate as much of this process as possible while maintaining the required security controls at the network boundary.

SecuNexa Pipeline Architecture

SecuNexa was purpose-built for air-gapped deployment. The entire platform, including SecuTrace (SAST), SecuLens (SCA), SecuStrike (DAST), and the unified vulnerability management dashboard, installs and operates without any internet connectivity. Licensing uses air-gapped activation that does not require periodic check-ins.

SecuLens maintains a local vulnerability database that is updated through signed, encrypted update packages transferred via the organization's approved media process. The update packages are compact (typically under 50 MB) and include incremental CVE data, updated license information, and enhanced detection signatures.

CI/CD integration is provided through command-line scanners that work with any CI server (Jenkins, GitLab CI, TeamCity, Azure DevOps Server) and produce standardized output formats (SARIF, CycloneDX, JUnit XML) that integrate with existing workflows. Quality gates are configurable per project, allowing different teams to adopt security testing gradually while maintaining organization-wide minimum standards.
