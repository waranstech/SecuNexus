Open-Source License Compliance — A Legal Guide for Indian Companies

SecuNexa Security Research Team
Share:

Indian software companies are among the largest consumers of open-source software in the world. From startups building products on open-source frameworks to large enterprises running critical infrastructure on open-source platforms, the Indian technology ecosystem depends heavily on community-developed software. Yet the legal obligations that accompany open-source usage remain poorly understood across much of the industry.

Open-source licenses are not simply permissions to use code freely. They are legally binding agreements that impose specific obligations on the users and distributors of the licensed software. Failure to comply with these obligations can result in license termination (losing the right to use the software), copyright infringement litigation, forced disclosure of proprietary source code, and financial penalties. For Indian companies preparing for IPO, undergoing due diligence for acquisition, or bidding on government contracts, license compliance gaps can derail business-critical transactions.

Understanding License Categories

Open-source licenses fall into two broad categories: permissive licenses and copyleft licenses. The distinction between these categories determines the obligations that flow to your organization when you use the licensed software.

Permissive licenses such as MIT, Apache 2.0, BSD (2-clause and 3-clause), and ISC impose minimal obligations. Generally, they require attribution (keeping the copyright notice and license text with the software) and include a warranty disclaimer. Permissive licenses allow you to use the licensed code in proprietary products, modify it without disclosing your changes, and distribute the combined work under any license you choose. The Apache 2.0 license adds explicit patent grants and a patent retaliation clause, making it somewhat more protective for both licensors and licensees.

Copyleft licenses such as GPL (versions 2 and 3), LGPL, AGPL, MPL, and EPL impose reciprocal obligations. The core concept of copyleft is that derivative works must be distributed under the same license as the original. This means if you modify GPL-licensed code and distribute the modified version, you must make your modifications available under the GPL. The strength of the copyleft obligation varies by license. Strong copyleft licenses like GPL v3 and AGPL extend the obligation broadly to the combined work. Weak copyleft licenses like LGPL and MPL limit the obligation to the modified files of the original library while allowing the larger work to remain proprietary.

The AGPL (Affero General Public License) deserves special attention because it extends copyleft obligations to network interaction. If you modify AGPL-licensed software and make it available as a service over a network (SaaS), users must be able to obtain the modified source code. This is particularly relevant for Indian SaaS companies and cloud service providers who may be incorporating AGPL-licensed components without realizing the disclosure obligations.

GPL Contamination and Its Implications

GPL contamination occurs when GPL-licensed code is incorporated into a proprietary application in a way that triggers the copyleft obligation, potentially requiring the entire application to be released under the GPL. This is not merely a theoretical risk. It has been the basis for numerous enforcement actions by copyright holders and organizations like the Software Freedom Conservancy.

The mechanism of contamination depends on how the GPL code is integrated. Static linking of a GPL library with proprietary code almost certainly creates a derivative work, triggering the copyleft obligation across the combined binary. Dynamic linking is more ambiguous but generally considered to create a combined work under GPL v2, and explicitly addressed in GPL v3. Copying GPL-licensed source files into a proprietary codebase clearly triggers the obligation. Even using GPL-licensed header files can create obligations depending on the substantiality of the copied content.

For Indian companies, GPL contamination often occurs inadvertently. A developer copies a utility function from a GPL-licensed project. A dependency resolution pulls in a GPL-licensed transitive dependency. A vendor delivers code that includes GPL components without disclosing this to the purchasing organization. In each case, the company inherits obligations it may not be aware of until a legal review or due diligence process uncovers the issue.

The consequences are severe. If GPL contamination is discovered during due diligence for an acquisition, it can reduce the company valuation or kill the deal entirely. If discovered by the copyright holder, it can result in litigation demanding either GPL compliance (releasing your proprietary code) or removal of the infringing code (which may require significant re-engineering).

License Conflicts and Incompatibilities

License conflicts arise when an application includes components under licenses with incompatible terms. The most common conflict occurs between Apache 2.0 and GPL v2. Apache 2.0 includes a patent retaliation clause that GPL v2 does not recognize, making the licenses incompatible in one direction: you cannot distribute Apache 2.0-licensed code under GPL v2 terms. This conflict was resolved in GPL v3, which is compatible with Apache 2.0.

Other common conflicts include combining code under different copyleft licenses (GPL v2 and MPL, for example), combining licenses with conflicting attribution requirements, and including code under licenses with incompatible patent provisions. When conflicts exist, there is no valid license under which the combined work can be distributed, potentially blocking distribution of your product.

Indian enterprises often discover license conflicts late in the development cycle because dependency trees in modern applications are deep and complex. A Java application might directly depend on 50 libraries, but the transitive dependency tree can easily include 500 or more components, each with its own license. Manually reviewing every license in this tree is impractical, making automated license analysis essential.

The Problem with Declared License Metadata

A critical challenge in license compliance is the unreliability of declared license metadata. Package managers like npm, Maven, and PyPI allow package authors to declare a license in their package manifest. However, these declarations are frequently inaccurate. Common issues include packages that declare no license at all (making them technically "all rights reserved"), packages that declare one license in metadata but include a different license in the actual license file, packages that contain code from multiple sources under different licenses but declare only one, and forked packages that change the declared license without having the legal right to do so.

Research has shown that license metadata discrepancies affect a significant percentage of packages in major ecosystems. Relying solely on declared metadata for compliance decisions creates material legal risk because the actual license obligations are determined by the license text included with the code, not by metadata in a manifest file.

Forensic License Detection

Forensic license detection addresses the metadata reliability problem by examining the actual source code for license evidence. Instead of trusting what a package claims its license is, forensic detection analyzes the source files for license headers, copyright notices, license text files, and code patterns that match known licensed components.

This approach catches discrepancies that metadata-only analysis misses. When a package declares an MIT license but contains files with GPL license headers, forensic detection flags the conflict. When vendored code includes copyright notices from a different project under a different license, forensic detection identifies the true obligations. When code has been copied from a licensed source without proper attribution, forensic detection recognizes the matching patterns.

For Indian companies undergoing M&A due diligence, pursuing IPO, or responding to license compliance audits, forensic evidence carries significantly more legal weight than metadata declarations. It demonstrates that the organization has conducted thorough analysis rather than accepting surface-level claims at face value.

Building a License Compliance Program

Effective license compliance requires a systematic program that combines policy, tooling, and process. Start by establishing an approved license policy that categorizes licenses into three tiers: approved for unrestricted use (typically permissive licenses), approved with conditions (weak copyleft licenses with specific usage guidelines), and prohibited (strong copyleft licenses or licenses incompatible with your business model). This policy should be reviewed by legal counsel familiar with Indian IP law and international open-source license jurisprudence.

Integrate automated license scanning into your CI/CD pipeline using an SCA tool with forensic detection capabilities. Every build should produce a license analysis report identifying all components, their licenses, any conflicts, and any policy violations. Block builds that introduce prohibited licenses or unresolved conflicts from progressing to production.

Maintain an exception process for cases where business requirements necessitate using components under restricted licenses. Each exception should be reviewed by legal and engineering leadership, documented with a risk assessment, and tracked for ongoing monitoring. Exceptions should have expiration dates and be re-reviewed periodically.
