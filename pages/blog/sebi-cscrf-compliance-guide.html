SEBI CSCRF 2024-2025: Complete Application Security Compliance Guide

SecuNexa Security Research Team
Share:

The Securities and Exchange Board of India (SEBI) released the Cyber Security and Cyber Resilience Framework (CSCRF) to establish a comprehensive cybersecurity baseline for all regulated entities in India's securities market. This framework applies to stock exchanges, clearing corporations, depositories, mutual funds, asset management companies, stockbrokers, portfolio managers, and other registered intermediaries. For application security teams, understanding and implementing the CSCRF requirements is now a critical compliance obligation.

This guide breaks down every application security-relevant requirement in the SEBI CSCRF, provides practical implementation guidance, and outlines how to build a compliance program that satisfies auditors while actually improving your security posture.

Understanding the SEBI CSCRF Scope

The CSCRF categorizes regulated entities into different tiers based on their criticality to the market infrastructure. Market Infrastructure Institutions (MIIs) such as stock exchanges and clearing corporations face the most stringent requirements, while smaller registered intermediaries have scaled requirements. However, application security mandates apply across all tiers, with variations in the frequency and depth of testing required.

The framework builds on earlier SEBI circulars on cybersecurity but consolidates and strengthens the requirements into a unified structure. It addresses six primary areas: governance, identification, protection, detection, response, and recovery. Application security requirements span multiple areas, primarily falling under protection and detection controls.

Importantly, the CSCRF does not exist in isolation. SEBI expects regulated entities to also comply with relevant provisions of the Information Technology Act 2000, CERT-In directives, and applicable data protection regulations. This means your application security program must address a composite compliance landscape.

Application Security Requirements in the CSCRF

The CSCRF mandates several specific application security controls. Secure Software Development Lifecycle (SSDLC) requirements direct organizations to integrate security into every phase of application development, from requirements gathering through deployment and maintenance. This includes threat modeling during the design phase, secure coding standards enforcement during development, security testing before release, and ongoing vulnerability monitoring in production.

Vulnerability Assessment and Penetration Testing (VAPT) must be conducted regularly for all critical applications. MIIs are required to perform VAPT at least quarterly, while other regulated entities must conduct assessments at minimum annually. The VAPT scope must cover both internal and external-facing applications, including web applications, APIs, mobile applications, and thick-client applications used for trading or back-office operations.

The framework requires organizations to maintain a comprehensive software inventory that includes all third-party and open-source components. This effectively mandates Software Composition Analysis (SCA) and the generation of Software Bill of Materials (SBOM) for critical applications. Known vulnerabilities in these components must be tracked and remediated within defined timelines.

Source code review requirements mandate that critical applications undergo static code analysis before deployment. The framework specifies that automated tools should be used for this purpose, and that the tools should cover the OWASP Top 10 and CWE Top 25 vulnerability categories at minimum.

The CSCRF also addresses API security, requiring that all APIs used for trading, settlement, or data exchange implement proper authentication, authorization, rate limiting, and input validation. API security testing must be part of the regular VAPT cycle.

Building a CSCRF-Compliant Application Security Program

Building a compliant program requires a structured approach that addresses each requirement category while creating an efficient, maintainable process.

Start with establishing governance and policy. Create an Application Security Policy document that references the CSCRF requirements, defines roles and responsibilities, specifies testing frequency for each application tier, establishes vulnerability remediation SLAs, and outlines the exception and risk acceptance process. This policy should be approved by the CISO and reviewed annually.

Next, implement a software asset inventory. Catalog every application within scope, including its technology stack, third-party dependencies, deployment environment, data classification, and criticality rating. This inventory forms the basis for determining testing requirements and prioritizing remediation efforts. Use SCA tools to automate dependency inventory and map it to known vulnerabilities.

Deploy automated security testing tools that cover the three essential disciplines. For SAST, select a tool that supports the programming languages in your portfolio and can detect the OWASP Top 10 and CWE Top 25 vulnerability categories. Integrate it into the CI/CD pipeline so that every code change is analyzed before deployment. For SCA, implement a tool that maintains a comprehensive vulnerability database and can generate SBOM output in standard formats like CycloneDX or SPDX. For DAST, deploy a scanner capable of testing web applications and APIs in staging environments before production deployment.

Establish a vulnerability management workflow that tracks findings from discovery through remediation and verification. Define remediation SLAs based on vulnerability severity: critical vulnerabilities should have a 48-hour remediation window, high-severity findings should be addressed within one week, and medium-severity issues should be resolved within 30 days. Low-severity findings should be tracked and addressed within 90 days. These timelines should align with your organization's risk appetite and the CSCRF's expectation of timely remediation.

Generating Audit-Ready Evidence

SEBI audits require demonstrable evidence of compliance, not just policy documentation. Your application security program should generate evidence artifacts automatically wherever possible.

For SSDLC compliance, maintain records of threat modeling sessions, security requirement specifications, secure coding training completion records, and code review sign-offs. Automated SAST integration in CI/CD provides a natural audit trail showing that every deployment underwent security analysis.

For VAPT compliance, retain complete scan reports with timestamps, severity distributions, and trend analysis. Track the remediation lifecycle for each finding, including the date discovered, date assigned, date remediated, and date verified. Your tools should generate these reports in formats that auditors can independently verify.

For software inventory compliance, generate SBOM documents for each critical application and maintain a historical record of vulnerability status over time. SCA tools should produce reports showing the complete dependency tree, known vulnerabilities at the time of each scan, and remediation actions taken.

For API security compliance, document the security controls implemented for each API endpoint, including authentication mechanisms, authorization policies, rate limiting configurations, and input validation rules. API security test results should demonstrate coverage of the OWASP API Security Top 10.

Common Compliance Pitfalls to Avoid

Several common mistakes can derail CSCRF compliance efforts. Treating security testing as a periodic event rather than a continuous process is the most frequent problem. The CSCRF expects ongoing security assessment, not just quarterly scans. Integrating security tools into the development pipeline ensures that testing happens with every code change, not just at audit time.

Another common pitfall is focusing on tool deployment without establishing remediation workflows. Running scans is only valuable if vulnerabilities are actually fixed. Without defined SLAs, ownership assignments, and tracking mechanisms, scan results accumulate without action, creating both security risk and audit findings.

Using cloud-based security tools without considering data sovereignty implications is increasingly problematic. SEBI expects regulated entities to maintain control over sensitive data, including source code and vulnerability data. Organizations should evaluate whether their security tools transmit data to external servers and whether this complies with their data governance requirements.

Finally, neglecting to maintain historical compliance records creates difficulties during audits. SEBI auditors may request evidence from previous quarters or years. Your compliance program should include data retention policies that preserve scan results, remediation records, and policy documents for at least five years.
