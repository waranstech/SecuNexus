CERT-In Compliance — Automated Vulnerability Management

The Indian Computer Emergency Response Team (CERT-In) mandates strict cybersecurity practices including 6-hour incident reporting, vulnerability disclosure, and continuous security monitoring. SecuNexa automates the technical controls required by CERT-In directives, ensuring your organization can detect, report, and remediate vulnerabilities within mandated timelines.

What CERT-In Requires: 7 Core Mandates

CERT-In's directives under the Information Technology Act apply to all service providers, intermediaries, data centers, government organizations, and corporate entities operating in India.

1. Mandatory Incident Reporting (6 Hours)

All cybersecurity incidents must be reported to CERT-In within 6 hours of detection. This includes data breaches, unauthorized access, malware attacks, website defacement, and phishing campaigns. SecuNexa's real-time detection and alerting helps organizations identify incidents immediately, enabling rapid reporting within the mandated window.

2. ICT System Log Retention

Organizations must maintain logs of all ICT systems for a rolling 180-day period within Indian jurisdiction. SecuNexa maintains comprehensive scan logs, finding histories, and remediation records that contribute to the overall log retention mandate for application security activities.

3. Time Synchronization (NTP)

All systems must synchronize clocks with the NTP server of the National Informatics Centre (NIC) or the National Physical Laboratory (NPL). SecuNexa timestamps all scan results and findings using synchronized time sources, ensuring temporal accuracy of all compliance evidence.

4. KYC and Subscriber Verification

Service providers must maintain accurate subscriber and customer records. While SecuNexa does not handle KYC directly, it secures the applications that process, store, and manage subscriber data — preventing unauthorized access and data leakage from KYC databases.

5. Vulnerability Scanning and Assessment

Organizations must perform regular vulnerability assessments of their internet-facing assets, web applications, and internal systems. SecuNexa automates continuous vulnerability scanning across all application layers — source code, dependencies, runtime behavior, and API endpoints.

6. Designated Point of Contact

Every entity must appoint a designated Point of Contact (PoC) for CERT-In coordination. SecuNexa's compliance reporting module generates the technical evidence and incident data that the designated PoC needs for timely and accurate communication with CERT-In.

7. Cooperation with CERT-In Investigations

Organizations must provide information and assistance to CERT-In during investigations. SecuNexa's complete audit trail — including scan configurations, findings, remediation actions, and evidence artifacts — ensures all required information is readily available for regulatory inquiries.

CERT-In Directive Mapping to SecuNexa

How SecuNexa's four security engines map to specific CERT-In requirements for application-level compliance.

CERT-In Directive Requirement SecuNexa Engine Implementation Details
CERTIN-C1 Vulnerability Detection in Web Applications SecuStrike (DAST) Automated black-box scanning of web applications for OWASP Top 10, business logic flaws, and configuration vulnerabilities. Continuous or scheduled scans ensure timely detection.
CERTIN-C2 Source Code Security Review SecuTrace (SAST) Static analysis of application source code across 30+ languages. Detects injection flaws, authentication weaknesses, cryptographic misuse, and hard-coded secrets before deployment.
CERTIN-C3 Third-Party Component Vulnerability Tracking SecuLens (SCA) Continuous monitoring of all open-source dependencies against NVD, GitHub Advisory, and proprietary vulnerability databases. Alerts within minutes of new CVE disclosure affecting project dependencies.
CERTIN-C4 API Security Assessment SecuShield (API Security) Discovery and testing of REST, GraphQL, and SOAP APIs for authentication bypass, data exposure, injection, and rate limiting vulnerabilities. Validates API gateway security configurations.
CERTIN-C5 Incident Data Collection for Reporting SecuNexa Platform Comprehensive finding metadata including discovery timestamp, affected assets, vulnerability classification, CVSS score, and exploitation evidence — ready for CERT-In incident report format.
CERTIN-C6 Remediation Tracking and Verification SecuNexa Platform Tracks remediation status from detection through verification. Automated re-scanning confirms fixes are effective. SLA tracking ensures remediation within mandated timelines.
CERTIN-C7 Security Audit Trail Maintenance SecuNexa Compliance Module Complete audit trail of all security testing activities with immutable timestamps. 180+ day log retention for scan results, user actions, configuration changes, and compliance evidence.
CERTIN-C8 Sensitive Data Exposure Prevention SecuShield (API) + SecuTrace (SAST) Detects PII, Aadhaar numbers, PAN details, and financial data exposure in API responses, log outputs, error messages, and application source code to prevent data breaches reportable to CERT-In.

CERT-In Compliance Evidence Package

SecuNexa generates structured compliance artifacts covering all 21 CERT-In compliance attributes (CERTIN-C1 through CERTIN-C21), grouped into five categories for efficient audit preparation.

Vulnerability Inventory (C1–C5)

Complete catalog of all discovered vulnerabilities with CVE references, CVSS scores, affected assets, discovery timestamps, and exploitation likelihood assessments. Covers web, API, source code, and dependency findings.

Remediation Records (C6–C10)

Detailed remediation tracking including assigned owner, target SLA, fix implementation date, verification scan results, and closure confirmation. Demonstrates active vulnerability management to regulators.

Assessment Reports (C11–C15)

Periodic assessment summaries showing scan coverage, finding trends, risk posture changes, and comparative analysis across assessment cycles. Available in HTML, PDF, Excel, JSON, and CSV formats.

Compliance Matrix (C16–C18)

Control-by-control mapping showing CERT-In directive reference, SecuNexa coverage, evidence type, current compliance status, and assessment date. Machine-readable for GRC platform integration.

Audit Trail (C19–C21)

Immutable log of all security testing activities — scan initiations, configuration changes, user actions, report generation events, and system access records — maintained for 180+ days per CERT-In requirements.
